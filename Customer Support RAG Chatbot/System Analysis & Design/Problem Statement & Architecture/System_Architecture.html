Section 1 of 6 -- RAG-Powered Customer Support Chatbot

1 Problem Statement

Modern e-commerce platforms face significant challenges in delivering efficient, accurate, and timely customer support at scale. Traditional customer service models rely heavily on human agents to handle a large volume of repetitive queries, leading to operational bottlenecks and inconsistent user experiences.

1.1 Core Problem

Customer support systems in the e-commerce domain are unable to provide real-time, accurate, and contextually relevant responses to user inquiries without heavy reliance on manual intervention. This results in:

  • Delayed response times causing customer dissatisfaction and reduced loyalty.
  • Inconsistent answer quality across different support agents and shifts.
  • High operational costs due to reliance on large human support teams.
  • Inability to scale during peak traffic periods without proportional cost increases.
  • Loss of contextual awareness when customers reference previous messages in a conversation.

1.2 Proposed Solution

This project addresses these challenges through the development of a RAG-Powered Customer Support Chatbot that combines semantic search with Large Language Model (LLM) inference, enabling intelligent, context-aware, and domain-specific automated responses.

The system leverages Retrieval-Augmented Generation (RAG) with an Adaptive Pipeline that intelligently routes queries based on intent classification and confidence scoring, ensuring optimal response quality while minimising unnecessary computation.

3.1 Use Case Descriptions

Use CaseActorDescription
UC-1: Submit Text QueryCustomerThe user submits a natural language question through the chat interface. The Adaptive RAG pipeline classifies the intent, retrieves relevant context if needed, generates an answer, and applies confidence-based verification before returning the response.
UC-2: Submit Voice QueryCustomerThe user submits an audio recording. The system transcribes it using Whisper (STT), processes it through the same Adaptive RAG pipeline, generates a text answer, converts it to speech using ElevenLabs (TTS), and returns both text and audio responses.
UC-3: Authenticate via GoogleCustomer, GoogleThe user initiates login through Google OAuth 2.0. Upon successful authentication, the system creates or retrieves the user account and issues a JWT token for subsequent API requests.
UC-4: Provide FeedbackCustomerThe user submits positive (thumbs up) or negative (thumbs down) feedback on a chatbot response. Feedback is persisted in the database and logged to MLflow and Prometheus for quality tracking.
UC-5: View HistoryCustomerThe user retrieves their past conversations and individual messages, enabling continuity across sessions.
UC-6: Monitor PerformanceAdministratorThe administrator accesses the Streamlit dashboard to view real-time KPIs including response time, request throughput, retrieval confidence distribution, and error rates sourced from Prometheus.
UC-7: Evaluate ModelAdministratorThe administrator reviews MLflow experiment logs to analyse model performance, compare configurations, and identify areas for accuracy improvement.
UC-8: Review FeedbackAdministratorThe administrator reviews aggregated user feedback data through the dashboard to assess overall system quality and user satisfaction.

5 Adaptive RAG Pipeline

The core innovation of this system is the Adaptive RAG Pipeline, which intelligently routes queries through different processing paths based on intent classification and retrieval confidence scoring.

5.2 Pipeline Paths

PathConditionBehaviourExample
CasualRouter classifies intent as casualSkips retrieval entirely; LLM generates a direct conversational response."Hello, how are you?"
High ConfidenceRetrieval confidence >= 0.5Returns the generated answer directly without additional verification."Can you resend the verification link?"
Medium ConfidenceConfidence between 0.2 and 0.5Returns the generated answer; confidence is logged for monitoring."How do I return a product?"
Low Confidence (Verified)Confidence < 0.2, verification passesA secondary LLM check confirms the answer is supported by the retrieved context.Edge-case domain questions
Low Confidence (Rejected)Confidence < 0.2, verification failsThe system returns a safe fallback message instead of a potentially hallucinated answer."Do you accept Bitcoin payments?"

5.3 Key Design Decisions

  • Query Rewriting: When conversation history exists, the system rewrites vague follow-up queries (e.g., "What about that?") into standalone questions before retrieval, improving search accuracy.
  • Confidence Formula: confidence = 1 - L2_distance, where L2 distance is the Euclidean distance between the query embedding and the closest stored chunk in ChromaDB.
  • Conversation History: The system loads the last 6 messages from the active conversation to provide contextual awareness across interactions.
  • Rate Limiting: Critical endpoints are protected with SlowAPI: 10 requests/minute on /chat/ask, 5 requests/minute on /chat/voice.
