2 Sequence Diagrams

2.1 Text Query Flow (Support Intent)

This diagram shows the complete interaction flow when a customer submits a support-related text query through the /chat/ask endpoint.

sequenceDiagram
    actor Customer
    participant FE as Next.js Frontend
    participant API as FastAPI /chat/ask
    participant RL as Rate Limiter
    participant Router as LLM Router
    participant Gen as Generator
    participant Ret as Retriever
    participant Emb as Embedder
    participant VDB as ChromaDB
    participant LLM as Groq LLM API
    participant DB as PostgreSQL
    participant Prom as Prometheus

    Customer->>FE: Submit Question
    FE->>API: POST /chat/ask (JWT + question)
    API->>RL: Check Rate Limit (10/min)
    RL-->>API: Allowed

    API->>DB: Load Conversation History (last 6 msgs)
    DB-->>API: History Messages

    API->>Router: classify(question)
    Router->>LLM: Classify Intent
    LLM-->>Router: "SUPPORT"
    Router-->>API: intent = SUPPORT

    API->>Gen: rewrite_query(question, history)
    Gen->>LLM: Rewrite if vague
    LLM-->>Gen: Rewritten Question
    Gen-->>API: Clear Question

    API->>Ret: retrieve(question, n=3)
    Ret->>Emb: Embed Question
    Emb-->>Ret: 384-dim Vector
    Ret->>VDB: query(vector, top_3)
    VDB-->>Ret: Matching Chunks + Distances
    Ret-->>API: Results + Confidence

    API->>Gen: generate_answer(question, chunks, history)
    Gen->>LLM: Prompt + Context + History
    LLM-->>Gen: Generated Answer
    Gen-->>API: Answer Text

    Note over API: Confidence >= 0.5 → Return directly

    API->>DB: Save ChatMessage
    API->>Prom: Record Metrics
    API-->>FE: JSON Response
    FE-->>Customer: Display Answer
Figure 3: Sequence Diagram -- Text Query Flow (Support Intent)

    TTS-->>API: MP3 Audio Bytes

    API->>API: Base64 Encode Audio
    API->>DB: Save ChatMessage

    API-->>FE: JSON (answer + audio_base64)
    FE-->>Customer: Display Text + Play Audio

3 Activity Diagram

The activity diagram visualises the complete workflow of the Adaptive RAG Pipeline, showing all decision points and processing paths.

flowchart TD
    A([User Submits Query]) --> B{Input Type?}

    B -->|Text| D[Validate Input]
    B -->|Audio| C[Transcribe with Whisper]
    C --> D

    D --> E[Load Conversation History - Last 6 Messages]
    E --> F{LLM Router: Classify Intent}

    F -->|CASUAL| G[Generator: Direct Answer - No Retrieval]
    F -->|SUPPORT| H{Has Conversation History?}

    H -->|Yes| I[Rewrite Query for Clarity]
    H -->|No| J[Use Original Query]
    I --> K[Semantic Search in ChromaDB - Top 3]
    J --> K

    K --> L[Generator: Generate Answer with Context]
    L --> M{Confidence Level?}

    M -->|"HIGH >= 0.5"| N[Accept Answer]
    M -->|"MEDIUM 0.2-0.5"| N
    M -->|"LOW < 0.2"| O{Verify Answer with LLM}

    O -->|VALID| P[Accept Verified Answer]
    O -->|INVALID| Q[Return Fallback Message]

    G --> R[Save Message to PostgreSQL]
    N --> R
    P --> R
    Q --> R

    R --> S[Log Metrics to Prometheus + MLflow]
    S --> T{Voice Endpoint?}

    T -->|Yes| U[Generate Audio via ElevenLabs TTS]
    U --> V[Encode Audio as Base64]
    V --> W([Return JSON: Text + Audio])
    T -->|No| X([Return JSON: Text Only])

4 State Diagram

The state diagram represents the different states of a user query as it transitions through the system processing pipeline.

stateDiagram-v2
    [*] --> Received : User submits query

    Received --> Transcribing : Audio input
    Received --> Validating : Text input
    Transcribing --> Validating : Transcription complete

    Validating --> Rejected : Empty or invalid
    Validating --> Classifying : Input valid

    Classifying --> CasualProcessing : Intent = CASUAL
    Classifying --> SupportProcessing : Intent = SUPPORT

    CasualProcessing --> Generating : Direct LLM call

    state SupportProcessing {
        [*] --> QueryRewriting
        QueryRewriting --> Retrieving : Rewritten query
        Retrieving --> ContextFound : Chunks retrieved
    }

    ContextFound --> Generating : Context ready

    Generating --> ConfidenceCheck : Answer generated

    ConfidenceCheck --> Verified : Confidence HIGH or MEDIUM
    ConfidenceCheck --> Verifying : Confidence LOW

    Verifying --> Verified : Verification VALID
    Verifying --> FallbackGenerated : Verification INVALID

    Verified --> Persisting : Save to database
    FallbackGenerated --> Persisting : Save fallback

    Persisting --> AudioGenerating : Voice endpoint
    Persisting --> Completed : Text endpoint

    AudioGenerating --> Completed : Audio encoded

    Completed --> [*]
    Rejected --> [*]
Figure 7: State Diagram -- User Query Lifecycle

    class LLMRouter {
        -Groq client
        -String model

        -Groq client

        +generate_answer(question, chunks, history) str

        +direct_answer(question, history) str
        +verify_answer(question, answer, context) Dict

    class Retriever {

    class Embedder {
        -SentenceTransformer model
        -String model_name
        +embed_text(text: str) List
        +embed_texts(texts: List) ndarray

        -PersistentClient client
        -Collection collection

    class Transcriber {
