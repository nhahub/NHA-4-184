1 Database Overview

The system uses two separate data stores, each optimised for its specific purpose:

  • PostgreSQL (Relational Database): Stores structured data including user accounts, conversations, chat messages, feedback, and OTP codes. Managed through SQLAlchemy ORM with 5 normalised tables.
  • ChromaDB (Vector Database): Stores pre-computed vector embeddings of the knowledge base (QA chunks). Used exclusively for semantic similarity search during the retrieval stage of the RAG pipeline.

This section focuses on the PostgreSQL relational database design. The ChromaDB vector store is a flat collection of embeddings and does not follow traditional relational modelling.

6 Normalisation Analysis

The database schema satisfies the requirements of Third Normal Form (3NF):

  • First Normal Form (1NF): All columns contain atomic values. There are no repeating groups or multi-valued attributes.
  • Second Normal Form (2NF): All non-key attributes are fully functionally dependent on the entire primary key. No partial dependencies exist since all tables use single-column primary keys.
  • Third Normal Form (3NF): No transitive dependencies exist. All non-key attributes depend directly on the primary key and not on other non-key attributes.

Design Decision: The retrieved_context field in chat_messages stores the full text of retrieved chunks (denormalised) rather than referencing chunk IDs. This is intentional because ChromaDB chunks may be re-indexed or updated independently, and preserving the exact context used at query time is essential for auditability and debugging.

7 Vector Database (ChromaDB)

ChromaDB stores the knowledge base as vector embeddings for semantic similarity search. It operates separately from PostgreSQL and does not follow relational modelling.
